# Database Skill — PostgreSQL, SQLAlchemy, Alembic

Scope: `backend/packages/db`. PostgreSQL (in Docker), SQLAlchemy ORM, Alembic migrations.
Run all Alembic/pytest commands from `backend/`.

## Structure

```
backend/packages/db/
  models/
    product.py        # products: id, title, commission_rate, metadata
    creator.py        # creators: id, handle/name
    sales.py          # sales_daily: date, product_id, creator_id, units, gmv, commission
    market.py         # market_trends: scraped external trend rows + captured_at
    ai_prompt.py      # ai_prompts: prompt text, schedule, enabled, created_at
    ai_run.py         # ai_runs: prompt_id, output, status, created_at
    scrape_run.py     # scrape_runs: audit (started/finished, rows, status, error)
  session.py          # async engine + session factory
  upserts.py          # idempotent upsert helpers
  alembic/            # env.py + versions/
  base.py             # declarative Base, naming conventions
```

## Conventions

- **One declarative `Base`** with a consistent naming convention for constraints/indexes
  (so Alembic autogenerate is stable).
- **Idempotent upserts** (`INSERT ... ON CONFLICT DO UPDATE`) keyed on natural keys
  (e.g. `(date, product_id, creator_id)` for `sales_daily`) — re-scraping a day must not
  duplicate rows.
- **Daily snapshots** retained for time-series queries (don't overwrite history; upsert
  by date).
- **Timestamps**: store UTC; `created_at` / `captured_at` on time-sensitive tables.
- **Migrations are the only way the schema changes.** Never hand-edit tables.

## Migration workflow (the agent runs these)

```
alembic revision --autogenerate -m "describe change"   # generate from model changes
# review the generated migration before applying
alembic upgrade head                                    # apply
```

- Always review autogenerated migrations before `upgrade` (autogenerate misses some
  things — server defaults, type changes).
- Postgres runs in Docker; reach it via SQLAlchemy or `docker exec`, not a host `psql`
  (the `psql` CLI is not installed and is not required).
- **Per project rule: never involve git** — migrations are versioned by Alembic only,
  not committed.
